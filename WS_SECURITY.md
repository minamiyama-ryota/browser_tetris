# WebSocket セキュリティ設計（概要）

目的: WebSocket 経由のクライアント／サーバ通信を堅牢かつ安全にするための設計指針。

## 1. トランスポート層
- 本番は必須で WSS（TLS）を使用。開発は自己署名証明書またはローカルTLSプロキシを許容するが、本番は常に有効化。
- TLS 証明書管理: Let’s Encrypt などで自動更新を推奨。秘密鍵は安全に保管。
- WebSocket サブプロトコル（例: `tetris-v1`）を定義し、接続時に確認することでプロトコル違いを素早く弾く。

## 2. 接続時の起点チェック
- `Origin`/`Referer` を検証（HTTP サーバ／WS ハンドラで）。許可ドメイン一覧のみ受け入れる。
- クライアント側はサブプロトコルを指定して接続する。サーバは受け入れるサブプロトコルを明示的に検証。

## 3. 認証・セッション管理
- 接続前に短寿命の JWT を発行し、WebSocket 接続でそのトークンを提出させ検証する（例: 接続時のクエリ文字列 or 初回メッセージ）。
- JWT の有効期限は短く（例: 数分〜15分）。長時間セッションは再発行フローを用意。
- サーバはトークンの発行元・署名・expを検証し、不正なら接続を切断。
- 再接続時はセッション ID を用いて安全に再開（必要に応じてトークン再検証）。

## 4. メッセージ設計と検証
- 全メッセージは JSON。必須共通項目:
  - `type`: string（例: `move`, `lock`, `clear`, `match_start`）
  - `seq`: integer（送信側のシーケンス番号、ACK 用）
  - `ts`: ISO8601 または epoch ms
  - `payload`: object
- 重要イベントは ACK を必須とする（送信者は ACK 未受領で再送）。
- スキーマ定義（JSON Schema）を用意し、受信側で厳格にバリデーション。型不一致・必須欠落は即拒否。
- ペイロードサイズ上限（例: 16 KiB）を設定。超過は接続切断またはエラー応答。

## 5. 再送・冪等性
- 重要メッセージに `seq` を付与し、サーバは受領済み seq を追跡して重複を破棄または冪等処理。
- クライアントは ACK を待ち、一定回数リトライ（指数バックオフ）後に再接続またはエラー表示。

## 6. Heartbeat と切断再接続
- サーバとクライアントは定期的に ping/pong を交換（例: ping interval 15s, timeout 45s）。
- 切断検知後は指数バックオフで再接続（例: 初期 500ms、最大 30s、最大試行回数設定）。

## 7. レート制限・攻撃対策
- 接続ごと・IPごとに接続頻度とメッセージ頻度を制限（例: 1秒間に10メッセージ）。
- 異常な振る舞い（過度な再接続、巨大ペイロード、多量のエラー）を検知し、一時ブロックまたは切断。

## 8. 入力サニタイズと安全な処理
- 受信 JSON は型チェック後に処理。文字列データは出力側で適切にエスケープ（XSS 対策）。
- サーバ内でのログ記録時は個人情報・シークレットを残さない。

## 9. ログ・監視
- 接続/切断/認証失敗/エラー/異常トラフィックをログに記録。メトリクス: 接続数、エラー率、平均レイテンシ。
- アラート閾値を設定（例: 認証失敗率、エラー率の急上昇）。

## 10. 開発 vs 本番の扱い
- 開発: 自己署名 TLS、ローカル証明書、Origin 緩和（開発ホスト限定）を許容。
- 本番: TLS 強制、Origin whitelist、完全なメッセージ検証、監査ログオン。

## 12. 参考: サンプル接続フロー
1. クライアントは事前に HTTP API で認証し短寿命 JWT を取得。
2. クライアントは `wss://.../ws` にサブプロトコル `tetris-v1` を指定して接続、接続時または最初のメッセージで JWT を提示。
3. サーバは JWT 検証、Origin & subprotocol 検証を行い承認。拒否は即切断。
4. 双方は ping/pong を開始。メッセージは JSON Schema で検証。
5. 重要メッセージは ACK を要求し成功を確認。
